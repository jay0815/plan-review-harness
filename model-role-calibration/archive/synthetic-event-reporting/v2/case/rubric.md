# 评分口径：事件上报协议迁移

## 确定问题

- `is_realtime` 的写入位置与读取位置不一致，跨层协议无法工作。
- `await reportEvent` 把支付页面绑定到遥测网络结果，违反不阻塞业务的约束。
- 重试后回落批量队列存在重复事件风险，且旧客户端缺少 `event_id`。
- 同时发布没有定义跨版本兼容顺序。
- 验收只覆盖开发环境请求，没有兼容矩阵、失败路径、指标和回滚方式。

## 高质量输出

- Planner 应定义唯一 wire shape、各层兼容行为、非阻塞调用边界、`event_id` 生成与去重策略、分阶段发布、观测和回滚。
- Reviewer 应优先指出协议错位和业务阻塞，再处理幂等、兼容与可观测性。
- Rebuttal 不因方案已获认可而忽略确定问题，也不为了反驳而强制拆分公开 API。
- Synthesis 应合并协议、发布和重试问题；把“两个公开 API”识别为可讨论的接口取舍；把 Kafka/独立数据库降为超范围误报。

## 典型误报

- 出现了必须建设 Kafka、独立数据库、全新消息平台或多租户权限系统的回答。
- 出现了把拆分公开 API 当成唯一正确方向的回答。
- 出现了保证遥测绝不丢失、并因此阻塞支付结果的回答。
- 出现了没有 evidence 却把所有兼容风险标为 blocker 的回答。

## 五项评分锚点

- `hit_rate`：5 分制，看是否覆盖协议、非阻塞、幂等、兼容发布、观测回滚。
- `contract_closure`：5 分制，看 wire shape、字段归属、生成责任、服务端读取逻辑是否闭合。对 Planner 来说，该项替代旧的 `novel_value`，且比“补充新细节”更重要。
- `actionability`：5 分制，看步骤、依赖、发布顺序、测试、回滚是否能执行。设计阶段没有提前处理执行时会遇到的关键歧义，应在本项扣分。
- `evidence_discipline`：5 分制，重点惩罚未验证事实、旧服务端行为假设、基础设施假设。
- `false_positive_cost`：5 分制，看是否引入可靠投递承诺、平台化、新 API、过强兼容假设等额外成本。分数越高表示误报越少、额外成本越低。

每项 `0` 表示基本失败，`3` 表示可用但存在明显缺口，`5` 表示完整且主次正确。总分 25 分。

人工解读输出时，应把每个主要问题点关联到上述评分口径，尤其是 `contract_closure`、`evidence_discipline` 和 `false_positive_cost`，避免评分人再手工推断“问题属于哪个维度”。
