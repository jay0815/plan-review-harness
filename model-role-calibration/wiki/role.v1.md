# synthetic/event-reporting 角色评分 v1

## 适用范围

本文记录四个模型在 `synthetic/event-reporting` 单一案例中的角色表现。

- 模型：`kimi`、`deepseek`、`glm`、`qwen`
- 角色：Planner、Risk Reviewer、Architecture Reviewer、Execution Reviewer、Rebuttal Reviewer、Synthesizer
- 每个维度满分 5 分，总分 25 分
- 分数顺序：`hit_rate / contract_closure / actionability / evidence_discipline / false_positive_cost`
- `false_positive_cost` 越高，表示误报和额外工程成本越低

这些结论只用于当前移动端遥测协议案例，是后续多案例校准和角色筛选的 v1 基线，不代表全局模型排名。

## 角色建议总览

|---|---|---|---|---|

## Planner

### 最终评分

| 模型 | 五维评分 | 总分 | 角色定位 |
|---|---|---:|---|
| Qwen | `4 / 4 / 4 / 2 / 2` | 16 | 首选 |
| GLM | `4 / 4 / 4 / 1 / 2` | 15 | 备选、契约复核 |
| DeepSeek | `3 / 2 / 3 / 1 / 4` | 13 | 范围控制辅助 |
| Kimi | `3 / 2 / 2 / 1 / 1` | 9 | 暂不推荐 |

### 模型评价

**Qwen**

- 优点：覆盖协议、`event_id`、服务端去重、原生桥、分阶段发布、兼容测试、监控和回滚，整体最完整。
- 缺点：假设旧服务端会忽略未知字段；引入远程开关、存储和过强可靠性能力；线上报文结构仍不够唯一。

**GLM**

- 优点：契约闭环最强，能明确公共 API、线上报文、JS 生成责任、原生传输路径、发布顺序和测试。
- 缺点：大量假设旧服务端、MQ 和队列优先级能力；自行设计旧事件复合键去重，可能错误去重；范围有所膨胀。

**DeepSeek**

- 优点：范围控制最好，坚持支付流程非阻塞，主动排除至少一次投递、死信队列和平台化建设。
- 缺点：wire shape 不闭合；`event_id` 生成责任分散；对旧服务端兼容行为存在无依据假设。

**Kimi**

- 优点：覆盖复用 `reportEvent`、非阻塞、`event_id`、兼容和灰度等显性需求，实施步骤较具体。
- 缺点：多套协议和责任并存；引入新 API、本地持久化、重启重放和近似零丢失承诺；关键设计未收敛，无法直接执行。

### 推荐结论

使用 Qwen 作为主 Planner，GLM 负责契约复核，DeepSeek 负责范围控制。当前不建议 Kimi 独立承担 Planner。

## Risk Reviewer

### 最终评分

| 模型 | 五维评分 | 总分 | 角色定位 |
|---|---|---:|---|
| Qwen | `5 / 4 / 5 / 2 / 3` | 19 | 首选 |
| Kimi | `4 / 4 / 4 / 3 / 2` | 17 | 备选、辅助审查 |
| DeepSeek | `4 / 3 / 3 / 3 / 1` | 14 | 发散分析辅助 |
| GLM | `1 / 4 / 2 / 3 / 3` | 13 | 暂不推荐 |

### 模型评价

**Qwen**

- 优点：唯一完整覆盖字段协议、业务阻塞、幂等、跨版本发布、失败验收、生产观测和回滚。
- 缺点：脑补旧服务端 `404`、应用商店审核、重复发货和远程配置；事实纪律明显不足。

**Kimi**

- 优点：核心风险识别稳定，字段归属、跨版本契约和 `event_id` 生命周期意识较强。
- 缺点：越界设计本地状态机、页面生命周期和高优先级补发；错误地把批量降级本身判为需求失败。

**DeepSeek**

- 优点：风险分析深入，准确识别首次发送、重试和批量回落必须复用相同 `event_id`。
- 缺点：建议 JS 直连、替代发送路径和页面生命周期机制，范围扩张最严重；遗漏快速回滚能力。

**GLM**

- 优点：契约定位能力较好，能识别字段层级和 `event_id` 一致性问题。
- 缺点：识别阻塞后仍建议保留有限 `await`，并建设可靠投递能力，方向直接违反需求。

### 推荐结论

使用 Qwen 作为主 Risk Reviewer，但必须通过 prompt 强制“未知事实只能进入问题列表”。Kimi 可作为辅助审查。GLM 当前不适合该角色。

## Architecture Reviewer

### 最终评分

| 模型 | 五维评分 | 总分 | 角色定位 |
|---|---|---:|---|
| Qwen | `4 / 3 / 4 / 3 / 3` | 17 | 首选 |
| Kimi | `3 / 3 / 3 / 2 / 2` | 13 | 幂等与生命周期辅助 |
| DeepSeek | `3 / 2 / 3 / 3 / 2` | 13 | 依赖方向辅助 |
| GLM | `2 / 2 / 2 / 2 / 2` | 10 | 暂不推荐 |

### 模型评价

**Qwen**

- 优点：覆盖非阻塞边界、版本耦合、`event_id` 幂等和验收不足，整体最完整。
- 缺点：遗漏最直接的字段层级错位；引入 feature flag 和能力检测等未确认机制。

**Kimi**

- 优点：具备模块边界、发布耦合和稳定 `event_id` 生命周期意识。
- 缺点：错误否定实时失败后的批量降级；引入不匹配的生命周期、持久化和退避机制。

**DeepSeek**

- 优点：依赖方向和职责错位分析最好，明确重试策略不应改变支付页面展示时机。
- 缺点：用 API 重构和能力协商替代对现有 wire contract 的精确检查，容易过度设计。

**GLM**

- 优点：能够识别显性的业务阻塞和发布耦合。
- 缺点：问题重复，遗漏字段协议和幂等责任；部分建议会制造业务层与基础设施的新耦合。

### 推荐结论

使用 Qwen 作为主 Architecture Reviewer；Kimi 补充幂等生命周期检查，DeepSeek 补充依赖方向检查。GLM 当前不适合主审。

## Execution Reviewer

### 最终评分

| 模型 | 五维评分 | 总分 | 角色定位 |
|---|---|---:|---|
| Kimi | `5 / 4 / 5 / 3 / 3` | 20 | 首选 |
| DeepSeek | `4 / 5 / 4 / 3 / 3` | 19 | 备选、异步契约主审 |
| GLM | `3 / 3 / 4 / 3 / 3` | 16 | 交付物检查辅助 |
| Qwen | `3 / 2 / 3 / 2 / 2` | 12 | 暂不推荐 |

### 模型评价

**Kimi**

- 优点：覆盖最完整，可直接转化为 Planner 补充清单；能发现修改位置、接口、兼容、失败验收和回滚缺失。
- 缺点：越界设计状态机和远程配置；把未知旧版本行为写成确定影响。

**DeepSeek**

- 优点：执行契约最精确，独有地识别 `reportEvent` 返回 `void`、Promise resolve 语义和已有调用方行为变化。
- 缺点：保留短时等待例外；加入生命周期恢复和具体重试参数。

**GLM**

- 优点：擅长检查文件定位、团队任务、服务端职责、兼容矩阵和自动化测试缺口。
- 缺点：遗漏最关键的字段层级错误；把明确的非阻塞硬约束重新开放为待决事项。

**Qwen**

- 优点：能发现接口规范、调用位置和验收条件不可量化等执行信息缺口。
- 缺点：遗漏关键字段协议；`event_id` 责任判断自相矛盾；把专用接口、feature flag 和版本检测等未知机制变成执行要求。

### 推荐结论

使用 Kimi 作为主 Execution Reviewer，DeepSeek 负责异步契约复核，GLM 检查交付物和依赖完整性。当前不建议 Qwen 独立终审。

## Rebuttal Reviewer

### 最终评分

| 模型 | 五维评分 | 总分 | 角色定位 |
|---|---|---:|---|
| GLM | `4 / 4 / 4 / 3 / 3` | 18 | 首选 |
| Kimi | `4 / 4 / 4 / 3 / 3` | 18 | 备选 |
| Qwen | `4 / 4 / 3 / 2 / 2` | 15 | 辅助反驳 |
| DeepSeek | `3 / 3 / 3 / 2 / 2` | 13 | 暂不推荐终审 |

### 模型评价

**GLM**

- 优点：四者中最均衡，能够独立挑战专家共识并主动说明可能误报的成立条件。
- 缺点：依赖具体超时参数和旧版本行为假设；发布建议建立在未经验证的兼容行为上。

**Kimi**

- 优点：独立性和契约检查较强，能发现实时失败回落批量队列后的 Promise 结算语义缺口。
- 缺点：使用有限等待折中明确的非阻塞约束；推测财务风险和 BI 数据污染。

**Qwen**

- 优点：核心反驳完整，能区分“同步发布本身”和“缺少兼容措施”，并要求唯一 JSON path。
- 缺点：混淆遥测数据与支付业务事实；扩张本地持久化和版本协商；遗漏生产验收与回滚问题。

**DeepSeek**

- 优点：能够打破错误共识，关注旧原生桥静默降级时调用方无法感知实际路径。
- 缺点：遗漏字段层级错位；为了强化反驳而引入可靠投递、能力探测和替代通道。

### 推荐结论

使用 GLM 作为 Rebuttal 主审，Kimi 作为备选。Qwen 可提供辅助反驳。DeepSeek 当前不适合作为最终反驳裁决者。

## Synthesizer

### 最终评分

| 模型 | 五维评分 | 总分 | 角色定位 |
|---|---|---:|---|
| Kimi | `4 / 3 / 4 / 3 / 3` | 17 | 首选候选 |
| DeepSeek | `4 / 3 / 4 / 3 / 3` | 17 | 备选候选 |
| Qwen | `4 / 3 / 4 / 2 / 3` | 16 | 辅助合成 |
| GLM | `3 / 2 / 3 / 2 / 2` | 12 | 暂不推荐 |

### 模型评价

**Kimi**

- 优点：合并能力和修订清单较强，能将发布耦合与接口、顺序和兼容矩阵合并，并识别 API 方向分歧。
- 缺点：没有区分确定字段协议错误与可选设计方向；加入旧客户端补丁和配置中心等来源未确认机制。

**DeepSeek**

- 优点：结构化合并最清楚，能识别拆分 API 需要修改原始需求前提。
- 缺点：把 Reviewer 未表态解释为共识；版本矩阵不够精确；加入无法作用于已发布旧客户端的修订方向。

**Qwen**

- 优点：覆盖和表达能力强，对 Kafka、独立数据库等误报的降权理由最充分，修订指令最完整。
- 缺点：来源归因错误；新增 CDN、feature flag 和 `Promise.race` 等来源中不存在的事实与机制。

**GLM**

- 优点：能够完成基本问题汇总并识别 Kafka 和独立数据库超出范围。
- 缺点：来源保真不足；错误分类 API 方向分歧；生成业务字段组合去重等可能破坏数据正确性的方案。

### 推荐结论

使用 Kimi 作为主 Synthesizer，DeepSeek 作为备选；Qwen 用于检查覆盖遗漏。当前不建议 GLM 独立承担合成裁决。

## v1 组合建议

若采用角色团队：

| 阶段 | 推荐模型 |
|---|---|
| Planner | Qwen，GLM 契约复核，DeepSeek 范围复核 |
| Risk | Qwen 主审，Kimi 辅助 |
| Architecture | Qwen 主审，Kimi / DeepSeek 专项补充 |
| Execution | Kimi 主审，DeepSeek 异步契约复核，GLM 交付物检查 |
| Rebuttal | GLM 主审，Kimi 备选 |
| Synthesis | Kimi 主合成，DeepSeek 备选，Qwen 覆盖复核 |

v2 需要验证：经过针对性 prompt 优化后，各角色是否仍需要上述 team 配置，还是优质 prompt 加单模型已经能够稳定满足角色要求。
