# LLM 五维评分记录：合成事件上报实施规划

## 范围

- 运行批次：`2026-06-11T06-18-10-701Z`
- 案例：`synthetic/event-reporting`
- 探针：`planner`
- 工程原型：移动端遥测协议迁移
- 模式：Claude Code 包装器 + `--with-json-validator`

本文只针对当前案例，不应被理解为通用的 LLM 排名。本场景的输出受移动端遥测协议设计问题影响，重点包括：`reportEvent`、支付成功场景的非阻塞行为、JS/原生/服务端跨层兼容、基于 `event_id` 去重，以及第一版范围控制。

## 当前提示词

````md
# 角色

你是一个实施规划者，负责把需求转化为可直接交给工程智能体或开发者执行的计划。

# 任务

请基于下面的需求与约束，产出一份决策完备的实施计划。

# 规则

- 明确目标、范围和不做事项。
- 明确关键接口、数据流、执行顺序和依赖。
- 覆盖失败处理、兼容性、回滚或恢复方式。
- 给出测试场景和可验证的验收标准。
- 不要假设可以读取未提供的仓库、文件或外部系统。
- 对输入中无法决定且会改变实现方向的问题，列入 `open_decisions`。
- 输出必须是 JSON。

# JSON 输出约定

- 最终回答必须是一个原始 JSON 对象，不要使用 Markdown 代码块。
- 字符串内部如果需要双引号，必须写成 `\"`，不要直接写未转义的 `"`。
- 不要输出尾逗号、注释、解释文字或模式定义之外的字段。
- 如果本次会话提供了 `validate_json_output` 工具，最终回答前必须先用完整候选 JSON 调用该工具。
- 只有当 `validate_json_output` 返回 `valid: true` 后，才可以把同一份 JSON 作为最终回答输出。

# 输入

# 需求：事件上报协议增加实时模式

现有移动端统一通过以下接口上报行为事件：

```ts
reportEvent(name: string, payload: Record<string, unknown>): Promise<void>
```

默认行为是先进入本地批量队列，满足数量或时间条件后上传。现在支付成功事件必须尽快上报，其他事件继续批量上报。

已知条件：

- JS、原生桥和服务端都由不同团队维护。
- 线上同时存在多个旧版本客户端和两个服务端版本。
- 当前服务端按 `event_id` 去重，但旧客户端没有生成 `event_id`。
- 上报失败不能阻塞支付成功页，也不能改变支付结果。
- 希望继续复用 `reportEvent`，不新增平行上报 API。
- 第一版只增加单事件实时模式，不建设新的消息平台、数据库或通用任务系统。

请给出可直接实施的计划。
````

## 已确认评分

评分采用修订后的五维五分制，总分上限为 25 分。

| 模型 | hit_rate | contract_closure | actionability | evidence_discipline | false_positive_cost | 总分 |
|---|---:|---:|---:|---:|---:|---:|
| qwen | 4 | 4 | 4 | 2 | 2 | 16 |
| glm | 4 | 4 | 4 | 1 | 2 | 15 |
| deepseek | 3 | 2 | 3 | 1 | 4 | 13 |
| kimi | 3 | 2 | 2 | 1 | 1 | 9 |

## 修订后的评分口径

- `hit_rate`：五分制。覆盖协议、非阻塞行为、幂等性、兼容发布、可观测性和回滚。
- `contract_closure`：五分制。在规划类评估中取代此前对 `novel_value` 的侧重。检查线上报文结构、字段归属、生成责任、桥接行为和服务端读取方式是否明确并形成闭环。
- `actionability`：五分制。检查实施步骤、依赖关系、发布过程、测试和回滚是否可执行。未解决的设计问题导致执行含糊时直接扣分。
- `evidence_discipline`：五分制。对未经验证的事实扣分，例如旧服务端如何处理未知字段、假定已有的基础设施、存储语义和旧事件去重能力。
- `false_positive_cost`：五分制。分数越高，额外成本越低。对可靠投递范围膨胀、平台化、新增公共 API、不必要的基础设施和过强的兼容性假设扣分。

## 模型表现

### qwen

优点：
- 整体命中率较好。
- 使用 `reportEvent(..., options)`，避免污染业务 `payload`。
- 覆盖 `event_id`、服务端去重、原生桥、功能开关、分阶段发布、兼容性测试和回滚。

问题：
- `evidence_discipline`：一方面假设 V1 服务端会忽略未知字段，另一方面又把该事项列为待决问题。
- `false_positive_cost`：通过重试和降级到批量队列的方式倾向于实现可靠遥测投递；验收表述暗示事件不能丢失。
- `contract_closure`：整体基本闭环，但规范报文封装和线上结构仍可更明确，JS 与原生层对实时通道的责任略有重叠。

### glm

优点：
- 本轮中契约闭环最强。
- 清晰区分公共 API 输入（`options.realtime`）与线上报文封装（`realtime`、`event_id`）。
- 明确 JS SDK 的生成责任和原生桥传输路径。
- 给出分阶段发布、集成测试、监控和回滚方案。

问题：
- `evidence_discipline`：假设旧服务端会忽略新增字段。
- `evidence_discipline`：假设已有 MQ 或队列优先级能力。
- `evidence_discipline`：为旧事件自行设计 `event_name + user_id + timestamp_window(60s)` 去重方式，可能造成错误去重。
- `false_positive_cost`：引入即时桥接方法、优先级路由、旧事件复合键去重和“最终到达服务端”语义，扩大了第一版范围。

### deepseek

优点：
- 对可靠投递范围膨胀的控制最好。
- 明确排除至少一次投递语义、重试队列、死信队列和平台化建设。
- 有力保障支付成功场景的非阻塞行为。

问题：
- `contract_closure`：最终线上报文结构不清晰。`config.realtime`、请求中的 `realtime=true` 和 `event_id` 位置没有统一到一个规范封装中。
- `contract_closure`：`event_id` 的生成责任分散在 JS SDK 和原生层。
- `evidence_discipline`：假设旧服务端会忽略未知的实时字段。
- `hit_rate`：线上报文结构的不确定性较严重，降低了覆盖质量。

### kimi

优点：
- 覆盖显性需求：复用 `reportEvent`、支付成功页非阻塞、`event_id`、兼容性和灰度发布。
- 在重试、原生行为和测试方面比部分输出更具体。

问题：
- `contract_closure`：把协议控制字段放进业务 `payload`，同时又引入原生实时 API 和请求头，没有唯一权威的线上报文结构。
- `evidence_discipline`：假设旧服务端会忽略未知字段，并假设新客户端请求旧服务端仍然安全。
- `evidence_discipline`：尽管第一版范围有限，仍提到 Kafka 或即时消费队列。
- `false_positive_cost`：可靠投递范围严重膨胀，包括重试、本地持久化、重启后重放、“保证最终到达”和接近零丢失。
- `actionability`：执行会被尚未解决的设计选择阻塞，而这些选择本应在实施步骤之前确定。

## 跨模型结论

共同优点：
- 大多数模型识别出需要复用 `reportEvent`。
- 大多数模型识别出支付成功页必须保持非阻塞。
- 大多数模型提到 `event_id`、去重、兼容性、灰度发布和回滚。

共同弱点：
- 证据纪律是最薄弱的维度。模型经常把未知事实转化为实施假设。
- 契约闭环程度不一致。模型经常未能定义贯穿 JS、原生桥和服务端的唯一规范线上报文结构。
- 即使案例要求的是移动端遥测尽力而为语义，模型仍容易引入后端可靠投递模式。
- 兼容性结论往往过强。“旧服务端会忽略未知字段”被反复无依据地假设。
- 重试或降级方案常在声称减少丢失的同时引入重复事件风险。

## 高价值提示词改进

### 1. 要求定义规范线上契约

预期收益：
- 提高 `contract_closure`。
- 提高 `hit_rate`，因为协议覆盖会变得具体。
- 提高 `actionability`，因为实施团队可以获得一份权威字段映射。

建议增加：

```md
必须在实施步骤之前定义规范线上契约。
包括：
- 公共 API 参数
- 内部封装字段
- 原生桥方法和参数
- 服务端读取字段
- 每个字段的归属方和生成方

同一个语义概念必须只有一个权威的线上位置。
如果 realtime 或 event_id 出现在多个位置，说明哪个位置是规范位置，以及其他位置为何属于派生值或被禁止使用。
```

### 2. 增加明确的事实纪律规则

预期收益：
- 提高 `evidence_discipline`。
- 减少不安全的兼容性结论。
- 让待决问题更清晰。

建议增加：

```md
不要假设输入中未说明的事实。
如果计划依赖以下任一事项，应将其放入 open_decisions，且不得将其作为主路径保证：
- 旧服务端会忽略未知字段
- 旧服务端接受新的封装结构
- 已有 MQ、Kafka、优先级队列、Redis 或持久化本地存储
- 批量队列在进程终止后仍能保留
- 没有 event_id 的旧事件可以安全去重
```

### 3. 将失败语义定义为尽力而为遥测

预期收益：
- 提高 `false_positive_cost`。
- 减少可靠投递范围膨胀。
- 更符合支付流程非阻塞要求。

建议增加：

```md
这是尽力而为的遥测，不是可靠投递系统。
硬性要求是上报失败不得阻塞或改变支付成功结果。
除非输入明确提供相关基础设施和要求，否则不得承诺零丢失、至少一次投递、重启后重放、死信队列或持久化重试队列。
如果提出重试或降级到批量队列，应说明重复风险和准确的幂等边界。
```

### 4. 强制提供兼容性矩阵

预期收益：
- 提高 `hit_rate`。
- 提高 `evidence_discipline`。
- 改善发布顺序设计。

建议增加：

```md
提供兼容性矩阵：
- 旧客户端 -> 旧服务端
- 旧客户端 -> 新服务端
- 新客户端 -> 旧服务端
- 新客户端 -> 新服务端

对每个组合说明：
- 预期行为
- 是否依赖未经验证的事实
- 重复或丢失风险
- 回滚行为
```

### 5. 对无充分理由放入业务载荷的控制字段扣分

预期收益：
- 提高 `contract_closure`。
- 避免再次出现原始的 `payload.is_realtime` 与顶层读取位置不一致问题。

建议增加：

```md
协议控制字段默认不应放入业务 payload。
如果把 realtime/event_id 放入 payload，需要说明 JS、原生桥和服务端如何读取同一个字段，以及如何避免污染业务 payload。
```

### 6. 要求在实施步骤之前列出执行阻塞项

预期收益：
- 提高 `actionability`。
- 在安排实施顺序之前暴露尚未解决的设计问题。

建议增加：

```md
在 implementation_steps 之前列出 "blocking_design_decisions"。
如果阻塞性决策影响字段归属、线上报文结构、旧服务端兼容性、去重语义或存储/重试行为，不得假装该事项已经确定后继续规划。
实施步骤可以包含针对阻塞项的验证步骤，但不得把未经验证的结果作为保证条件。
```

## 预期最大收益

1. 规范线上契约：对 `contract_closure` 和 `actionability` 的提升最大。
2. 事实纪律规则：对 `evidence_discipline` 的提升最大。
3. 尽力而为失败语义：对 `false_positive_cost` 的提升最大。
4. 兼容性矩阵：对 `hit_rate` 和发布规划有显著的次级提升。

下一版提示词应优先落实这四项改动，再考虑增加更多输出字段或更细的模式定义约束。
