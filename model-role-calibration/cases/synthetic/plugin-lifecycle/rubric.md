# 评分口径：CLI 插件资产生命周期

## 确定问题

- 覆盖和整目录删除会破坏用户修改及其他来源文件。
- 没有 manifest、版本、hash 或资产归属记录，无法判断状态和安全卸载。
- `--dry-run` 没有列出计划动作，不能承担预检职责。
- 失败后重跑没有处理部分写入，恢复与幂等性未定义。
- 验收只覆盖空目录 happy path，缺少重复执行、未知文件和 scope 隔离。

## 高质量输出

- Planner 应定义受管 manifest、逐文件归属和 hash、用户修改冲突策略、临时目录与原子提交、dry-run 输出、scope 路径解析和完整测试矩阵。
- Reviewer 应明确 CLI 可以管理 bundled 资产，但必须限制所有权边界。
- Execution 输出应给出命令行为、输入输出、失败状态和可验证验收。
- Synthesis 应保留资产所有权与恢复问题；把“必须在线 marketplace”和“灰度平台”识别为超范围建议。

## 典型误报

- 出现了认为本地 CLI 绝对不能安装 bundled 资产的回答。
- 出现了要求在线 marketplace、账号系统、灰度平台、SLA 或远程 kill switch 的回答。
- 出现了建议无条件覆盖用户修改或删除未知文件的回答。
- 出现了只讨论命令命名、目录命名等偏好问题的回答。

## 五项评分锚点

- `hit_rate`：5 分制，看是否覆盖所有权、manifest、冲突、恢复、dry-run 和测试。
- `contract_closure`：5 分制，看资产归属、manifest/hash、状态分类、安装/更新/卸载边界是否闭合。
- `actionability`：5 分制，看命令语义、数据结构、操作顺序、失败恢复和验收是否能执行。
- `evidence_discipline`：5 分制，看是否遵守本地文本资产和无后台服务的边界。
- `false_positive_cost`：5 分制，看是否避免 marketplace、账号系统、灰度平台和线上服务治理噪音。分数越高表示误报越少、额外成本越低。

每项 `0` 表示基本失败，`3` 表示可用但存在明显缺口，`5` 表示完整且主次正确。总分 25 分。
