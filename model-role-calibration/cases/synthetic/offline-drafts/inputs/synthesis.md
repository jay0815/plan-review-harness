# 需求背景

移动端离线草稿支持多设备编辑。服务端有 revision 和幂等提交能力；本地升级到 v2。第一版不做实时协同和字段级自动合并。

# Architecture Reviewer

- 仅用 `synced` 布尔值无法表达 dirty、syncing、conflict、failed 等状态，也没有记录基准 revision。
- 服务端 last-write-wins 最简单，应直接作为唯一冲突策略，不需要用户介入。
- 本地草稿应始终是 source of truth，服务端不能拒绝客户端上传。

# Execution Reviewer

- 缺少 v1 到 v2 的数据迁移规则；把所有旧草稿标为已同步可能让未上传内容永远丢失。
- 每次重试生成新幂等键会失去幂等效果；需要稳定的同步操作标识。
- 批量部分成功后本地统一保持未同步，会重复提交已成功记录。

# Risk Reviewer

- 没有基于 revision 的冲突检测，两台设备会静默覆盖。
- 应实现 CRDT，才能从根本上解决并发编辑。
- 需要保留本地内容、远端内容和基准版本，并在无法自动解决时进入用户冲突处理。
- 缺少迁移失败恢复、重试退避和同步状态可观测性。

# 合成任务

合并重复问题，识别真正的方向分歧，降权无 evidence 或超范围建议，并指出哪些决策必须由产品或用户裁决。
