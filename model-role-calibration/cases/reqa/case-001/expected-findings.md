# 期望发现

仅供人工评分使用。

本文件记录期望从模型输出中看到的高质量回答要点。

## 产品边界

- 需要明确指出 `reqa` 的产品定位应是文档抽取 CLI。
- 需要明确指出 `reqa` 的核心职责是：基于用户明确提供的 URL / selector 抽取文档内容，并生成本地可复核、可被 LLM 读取的产物。
- 需要明确指出 `reqa` 不应被设计成需求分析工具、规划准入工具、模型评审工具或多 agent 编排工具。
- 需要明确指出“文档抽取”和“判断需求是否足够进入开发规划”是两类不同职责，不应放在同一个 CLI 产品边界内。
- 需要明确指出 `reqa` 不应承担面向业务需求质量的最终判断，也不应输出规划准入结论。

## 能力边界

- 需要明确指出 `reqa` 应优先设计稳定的抽取链路，而不是设计需求分析、角色化审查、规划评估或 agent evaluation 能力。
- 需要明确指出面向 LLM 的支持应围绕“如何使用 CLI 和理解产物”，而不是让 `reqa` 内置分析流程。
- 需要明确指出如果需要需求分析、规划判断、模型评审或 agent evaluation，这些应属于上层 harness / workflow，而不是 `reqa`。
- 需要明确指出 `reqa` 的输出应该是文档产物和必要元信息，而不是分析报告、评分结论或开发计划。

## Skill 边界

- 需要明确指出 `reqa` 可以保留 skill 资产，因为 CLI 面向 LLM / agent 时需要清晰的使用说明。
- 需要明确指出 skill 的职责是指导 LLM / agent 正确使用 `reqa` CLI，而不是替 `reqa` 扩展出新的分析职责。
- 需要明确指出 skill 应说明：什么时候使用 `reqa`、如何调用 CLI、输出目录在哪里、各产物分别用于什么、失败时如何检查。
- 需要明确指出 skill 资产可以留在源码目录中，后续通过自定义 marketplace 或其他工程机制集成；`reqa` CLI 自身不应负责安装或分发 skill。
- 需要明确指出 agent 不应作为 `reqa` 的核心设计对象；agent 如果需要，应由上层系统定义。

## 工程基础

- 需要明确指出从头开始的 CLI 项目应在方案阶段设计基础工程 harness。
- 需要明确提到至少部分基础验证能力，例如 lint、format、typecheck、unit test framework。
- 需要明确指出这些基础验证能力是 CLI 可维护性的基础，不应等实现后再补。
- 需要明确指出文档抽取能力应有可回归验证方式，例如 HTML 到 Markdown 转换、输出路径生成、图片处理、错误处理等核心行为的测试。

## 好回答的特征

- 能把 `reqa` 的正确边界收敛为：文档抽取 CLI + 本地产物协议 + LLM 使用 skill。
- 能指出哪些能力不应该被设计进来：需求分析、规划准入、角色化审查、模型评审、多 agent 编排、agent evaluation。
- 能认可文档抽取能力本身的价值，而不是因为范围错误而全盘否定 CLI。
- 能保留安全边界：不读取 credential / storage / header，不做 crawling、scraping-at-scale、bypass、security testing。
- 能给出设计阶段的边界建议。
